# Sitemap & Crawler Fetchability Hardening

Google Search Console reported "Couldn't fetch" for
`https://requiems.xyz/sitemap.xml` while direct requests returned valid XML.

This design records the implementation that hardens crawler-facing sitemap
endpoints at the Rails layer (without requiring Caddy-specific behavior
changes).

---

## Problem Statement

Observed behavior during investigation:

1. `GET https://requiems.xyz/sitemap.xml` returned valid XML content.
2. Search Console still reported intermittent fetch failures.
3. Sitemap-related responses were missing explicit, crawler-oriented
   cache/content guarantees at the application level.

Even when proxy behavior is correct, crawler reliability improves when the
origin app itself provides unambiguous response headers for machine-consumed
endpoints.

---

## Goals

1. Ensure sitemap and LLM text endpoints are explicitly typed (`application/xml`
   or `text/plain`) from Rails.
2. Ensure crawler-facing endpoints are publicly cacheable with short TTL.
3. Validate behavior with controller-level tests.
4. Keep changes localized to Rails so behavior is stable even if proxy config
   changes.

---

## Non-Goals

1. Re-architecting Caddy/Cloudflare routing.
2. Changing sitemap XML structure/content generation logic.
3. Introducing long-lived cache durations.

---

## Design

### 1. Rails-Level Header Ownership

`SitemapController` now owns crawler-facing response metadata:

- Adds `expires_in 5.minutes, public: true` on:
  - `sitemap`
  - `llms`
  - `llms_full`
  - `api_doc`
- Adds explicit content type mapping via `before_action`:
  - `sitemap` -> `application/xml`
  - `llms`, `llms_full`, `api_doc` -> `text/plain`

This removes ambiguity and ensures consistent behavior from the origin app.

Implementation:
[apps/dashboard/app/controllers/sitemap_controller.rb](apps/dashboard/app/controllers/sitemap_controller.rb)

### 2. Regression Tests for Crawl-Critical Behavior

Added integration tests to lock behavior:

- `/sitemap.xml` responds success, XML content type, public cache-control
- `/llms.txt` responds success, plain text content type, public cache-control
- `/llms-full.txt` responds success, plain text content type

Implementation:
[apps/dashboard/test/controllers/sitemap_controller_test.rb](apps/dashboard/test/controllers/sitemap_controller_test.rb)

## Tradeoffs

### Why Rails-level instead of only Caddy-level?

1. The sitemap/LLM responses are generated by Rails; setting semantics at source
   is more robust.
2. Proxy/CDN config may evolve; origin guarantees remain stable.
3. Test coverage can enforce behavior directly in app CI.

### Why short public cache (300s)?

1. Helps crawler fetch stability and intermediary caching.
2. Keeps updates near-real-time for frequently changing docs/catalog content.
